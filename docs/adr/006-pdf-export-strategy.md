# ADR 006: PDF Export Strategy

## Status

Accepted

## Decision

The product provides two PDF/print paths:

- A WebView print flow that renders sanitized print HTML with page setup CSS.
- A lightweight generated PDF writer for deterministic text-oriented export.

The generated PDF writer paginates a bounded lightweight body projection using `word-core` page setup, emits one page object per generated page, honors explicit page breaks, repeats simple header/footer text, resolves page-number/page-count/date fields deterministically, and supports validated page ranges.

The body projection supports text lines, simple table rows/cells, and simple figures. Tables render as vector cell boxes with wrapped cell text. Image blocks render with rough alignment/scale handling from `ImageBlock` presentation metadata. Sprint 038 embeds only safe in-document baseline `image/jpeg` assets as bounded PDF image XObjects using `/DCTDecode` after stripping APP/COM JPEG metadata marker segments; unsupported payloads remain visible placeholder boxes with safe visible alt/caption text. The PDF stream does not emit image asset IDs, original source names, local paths, usernames, hostnames, APP/COM JPEG metadata, or PDF creation/producer metadata.

Sprint 037 adds bounded active PDF URI annotations for safe external `http`, `https`, and `mailto` inline text links in paragraphs, headings, list text, and table cell text. Link rectangles are approximate text-run areas produced by the lightweight layout pass, and annotation objects are capped per page and per exported document. Safe internal `#bookmark` links remain visible text only in generated PDFs.

JPEG embedding is capped at 32 images per generated PDF, 8 MiB per embedded JPEG, 8192 px per side, 20,000,000 pixels, and grayscale/RGB component counts. It uses a small bounded baseline JPEG structure parser and metadata stripper instead of a decoder or heavy image-processing dependency.

PNG/GIF/WebP PDF image embedding, progressive JPEG embedding, JPEG decoding, downsampling/recompression, crop/rotation, EXIF interpretation/selective preservation, color-management precision, rich PDF image metadata, merged cells, table resizing, rich table styling, formulas, complex nested table layout, embedded/subset fonts, internal PDF destinations, exact glyph-level link geometry, complex script shaping, page-bottom footnote layout, and editor-preview/full print-layout fidelity remain deferred until a layout-engine ADR is accepted.

## Consequences

- README must not claim full desktop-publishing PDF fidelity.
- PDF tests cover header/xref/EOF smoke, multi-page object structure, explicit page breaks, header/footer page fields, page-range validation, table border drawing and cell text, JPEG image XObject emission, APP/COM JPEG metadata stripping, unsupported/malformed/oversized image fallback privacy, image cap fallback, page-range behavior with images, image placeholder text/privacy, structured-block pagination, safe external URI annotations, annotation bounds, page-range behavior with annotations, and absence of private path/build metadata.
- Print HTML is generated by Rust from `word-core` and must share the sanitized HTML export boundary.
- The Tauri capability file grants only the narrow WebView print permission in addition to core defaults.
