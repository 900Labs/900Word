# ADR 005: Import And Export Strategy

## Status

Accepted

## Decision

Rust owns import/export. The initial native format is ODT, with TXT, sanitized HTML, print HTML, basic PDF export, and bounded DOCX conversion. DOCX is import/export only and must pass through `word-core`; it is not a native save format. Binary `.doc` is deferred.

Sprint 033 keeps that boundary while adding simple DOCX page-region conversion for default and first-page headers/footers plus page-number, page-count, and date fields. Sprint 034 extends the same conversion-only boundary to safe embedded DOCX image relationships under `word/media/`, using existing `word-core` assets and `ImageBlock`s. Sprint 035 extends it to simple anchored DOCX comments using existing `word-core` comment threads and inline comment anchors. Sprint 056 extends it to unique safe paragraph/heading bookmark markers plus generated 900Word TOC paragraphs using `Word900Toc*` styles and safe internal links. Sprint 057 extends it to bounded safe table-cell presentation tags for the existing 900Word light-fill, hidden-border, and simple text-alignment model. Sprint 058 extends it to bounded safe paragraph formatting tags for the existing 900Word alignment, automatic line-spacing, spacing, and indent model. Sprint 059 extends it to bounded safe inline run formatting tags for the existing 900Word direct mark, supported font-size, direct text-color, and safe highlight model. Sprint 060 extends it to bounded safe body-level section page setup tags for the existing 900Word page width, height, and margin model. Sprint 061 extends it to explicit top-level body paragraph page-break runs for the existing `PageBreak` block model. Sprint 062 extends it to truthy top-level body paragraph `w:pageBreakBefore` flags by normalizing them into existing `PageBreak` blocks while ignoring unsupported nested flags with generic warnings. Sprint 063 extends it to allowlisted DOCX run fonts by normalizing them into existing generic `word-core` font-family IDs. Sprint 064 extends it to safe direct DOCX run shading fills by normalizing them into the existing bounded highlight palette. Sprint 065 extends it to bounded generated square DOCX image extents by normalizing them into existing `word-core` image scale metadata. Sprint 066 extends it to generated 900Word image caption paragraphs by normalizing plain `Word900ImageCaption` paragraphs immediately following supported images into existing `word-core` image caption metadata. Sprint 067 extends it to simple left/center/right paragraph `w:jc` alignment around single supported image paragraphs by normalizing that value into existing `word-core` image alignment metadata. Sprint 068 extends it to supported DOCX drawing `descr`/`title` metadata by normalizing those values into existing `word-core` image alt text and exporting local alt text on generated drawing metadata. Header/footer, page setup, explicit page breaks, paragraph page-break-before import, image, generated image captions, simple generated image alignment, bounded image alt text, comments, bookmarks, generated TOC shapes, bounded cell presentation shapes, bounded paragraph formatting shapes, bounded inline run formatting shapes, generic font-family mappings, safe run-shading highlight imports, and generated square image scale extents are local package content only; remote targets, unsafe relationship targets, linked images, source image names, arbitrary image sizing/cropping/layout fidelity, image compression/downsampling, floating/wrapping/anchoring fidelity, Word-native caption fields, rich caption formatting, even-page regions, complex fields, complex section layouts, multi-section page layout, columns, header/footer distance fidelity, layout-generated page markers, column-break semantics, page breaks in unsupported contexts beyond visible spacing or warning-only ignore, hidden Word page-break-before export fidelity, comment replies/threading/resolved-state fidelity, tracked changes, notes, Word TOC field-code fidelity, deterministic TOC page numbers, arbitrary paragraph settings, inherited Word character styles, arbitrary font families, exact source font names, embedded fonts, theme fonts, theme colors, pattern run shading, arbitrary highlights, arbitrary table colors, per-side borders, rich table themes, broad media fidelity, full DOCX review fidelity, and full layout fidelity remain outside the accepted DOCX conversion scope.

## Consequences

- Imported HTML is sanitized before reaching the frontend.
- Exported HTML and print HTML are generated from `word-core`, not from raw imported HTML.
- Export-to-path commands validate format-specific extensions and return only format/byte-count summaries to the frontend.
- DOCX imports open as dirty unsaved documents so Save continues to mean ODT unless the user chooses Save As.
- External converters are not bundled in the bootstrap.
